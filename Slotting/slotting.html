Slotting Business Domain

Data Lineage & Operational Topology Report

I. Core Data Lineage (The "Hub" Topology)

At its foundation, the system relies on high-degree common primary and foreign keys that bridge isolated data tables. By mapping these, we identify the primary drivers of the slotting logic:

  • Primary Master Hub: The Item dimension acts as the most heavily interconnected node Degree: 4, structurally linking the CG_ABC, Change, FWP, and UPH_ABC data tables.
  • Operational Quantity Hub: The Open_CO_Qty metric Degree: 3 serves as a core connective tissue between CG_ABC, OpenTrips, and UPH_ABC.
  • Accuracy Hub: The Acc(%) field Degree: 3 creates a direct relationship across CG_ABC, ItemData, and UPH_ABC.
  • Infrastructure Hubs: Both Warehouse and Disabled function as tertiary connection points Degree: 3. Warehouse links the Change, FWP, and STO tables. Disabled binds Disabled_Loc, FWP, and ItemData.
  • Item Identity Cluster: Core item attributes like ITNBR, ITCLS, ITDSC, PICKPUT, and ITMCLSID uniformly share a connection degree of 3, acting as the structural backbone linking ItemData, Item_Query, and OpenTrips together.

II. Operational Execution Workflow

The underlying data is orchestrated via a sequential execution pipeline managed by the a99_Excute_FWP_ subroutine.

  • Initialization: The process begins by stripping all active auto-filters from the worksheets using the Filter_ module to ensure a clean data environment.
  • Sequential Staging: Data transformation follows a strict hierarchy, sequentially firing modules that build upon one another: it initializes base structures (a0_FWP_, a01_STO_), retrieves operational requirements (a1_Pull_Open_Trips_), and enriches the item data (a2_ItemData_, a3_fwp_add_columns_, a4_CG_ABC_, a5_ItemData_Class_).
  • Mapping & Metrics Calculation: The logic applies the physical location mapping (a6_Location_vs_Item_mapping_), calculates classification and family associations (a7_itemdata_get_location_abc_, a71_item_family_), and culminates in generating final performance outputs (a8_Metrics_).
  • Logging: Upon success, the system writes the completion timestamp alongside the calculated "Wall Time" onto Sheet9, permanently logging the update run.

III. Deep Dive: Location vs. Item Mapping Logic

The logic contained within a6_Location_vs_Item_mapping_ governs how physical warehouse infrastructure matches to specific inventory items.

  • Data Aggregation: The script first constructs a dictionary linking items to specific Forward Pick (FWP) locations parsed from Sheet3.
  • Attribute Gathering: It simultaneously extracts the "Item Type" and calculates total "OnHand" inventory volume from Sheet12.
  • Reconciliation Engine: The mapping applies rules directly to the primary Locations table (Sheet4). If an FWP location match exists, the item is mapped; otherwise, the space is actively flagged as z_empty fwp loc.
  • Exception Handling: The system attempts to assign the item type, defaulting to an alert status of z_Please Check if the source data is missing.
  • Inventory Placement: Finally, the exact mapped OnHand quantity is assigned to the location, inserting 0 where no inventory data is detected.

IV. Entity Data Dictionary Summary

The slotting domain relies on dense tables encapsulating specific business parameters:

  • ItemData (43 Dimensions): The most complex entity, maintaining comprehensive attributes such as physical weight constraints (ITMWEGHT), physical dimensions (PRDWIN, PRDHIN, PRDLIN), and pallet characteristics.
  • Locations (38 Dimensions): Manages physical layout and storage constraints, tracking capacities through metrics like Capacity Volume, Utilization_by_CBM, and specific aisle/tier orientations.
  • FWP (20 Dimensions): Manages forward-picking strategies by capturing fields like Replen Qty, Capacity Qty, Fwd Qty, and underlying utilization potentials.
  • Item_Query (36 Dimensions): An analytical snapshot capturing item metrics ranging from MOHTQ and QTYCR to specialized parameters like B2Z95S and open purchase orders (OPEN_PO).
